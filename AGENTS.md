# AGENTS.md

## Назначение документа

Этот документ фиксирует инженерные правила проекта:

- как организован код
- как устроены модули
- как работать с базой данных
- как оформлять DTO, контракты и REST API
- как писать тесты

Продуктовое описание, MVP и roadmap ведутся отдельно в [docs/product/mvp-crm-plan.md](/Users/viig/www/teachly/docs/product/mvp-crm-plan.md).

Репозиторий организован как monorepo:

- `backend/` содержит backend на Nest.js
- `frontend/` содержит frontend на Nuxt v4
- `docs/` содержит продуктовую и архитектурную документацию

---

## 1. Архитектурные решения

- frontend: Nuxt v4
- UI kit frontend: daisyUI
- backend: Nest.js
- база данных: PostgreSQL
- ORM: TypeORM
- API: REST
- паттерн доступа к данным: Repository Pattern
- шаблонизатор уведомлений: Handlebars

Проект должен быть изначально готов к мультиязычности:

- язык первой версии: русский
- архитектура не должна предполагать, что русский язык является единственным
- текстовые ресурсы интерфейса и системных сообщений должны проектироваться с возможностью локализации
- шаблоны уведомлений должны поддерживать хранение по языкам

Эти решения считаются зафиксированными. Изменение `TypeORM`, `REST` или базовой слоистой структуры требует отдельного архитектурного решения.

Для работы с актуальной документацией по фреймворкам, UI-библиотекам и tooling необходимо использовать MCP сервер `context7`.

Правила:

- для Nest.js, Nuxt.js, daisyUI, Tailwind CSS и других внешних библиотек или фреймворков сначала получать актуальную документацию через `context7`, а не полагаться на память модели
- при вопросах по API, конфигурации, composables, hooks, decorators, plugins, CLI и паттернам интеграции использовать `context7` как основной источник истины
- если в проекте используется сторонняя библиотека и есть риск, что API или рекомендации могли измениться, нужно сначала проверить документацию через `context7`

## 2. Слои backend

Каждый доменный модуль строится по трем слоям:

- controllers: маршрутный слой, guards, чтение входных данных, вызов сервиса, трансформация ответа в DTO
- services: бизнес-логика и прикладные сценарии
- repositories: инфраструктурный доступ к БД через `Repository<Entity>`

Жесткие правила:

- контроллеры не содержат бизнес-логику
- сервисы не работают с HTTP-деталями
- сервисы не пишут SQL напрямую
- репозитории не содержат бизнес-логику
- репозитории работают через API `Repository<Entity>`
- `queryBuilder` использовать только в крайних случаях, когда стандартного API репозитория недостаточно

## 3. Структура модулей

Каждый модуль организуется по доменному принципу:

```text
backend/src/{module}/
  {module}.module.ts
  constants/
    index.ts
  dao/
    {entity}.entity.ts
    migrations/
      {timestamp}-{migration-name}.ts
  controllers/
    {module}.controller.ts
  dto/
    {dto}.ts
  repositories/
    {module}.repository.ts
  services/
    {module}.service.ts
```

Правила:

- названия файлов и классов отражают домен
- если в модуле несколько контроллеров, сервисов или репозиториев, они остаются внутри тех же директорий
- миграции должны храниться по одному согласованному правилу во всем проекте

## 4. TypeORM и база данных

Сущности хранятся в `backend/src/{module}/dao`. Репозитории инжектят `Repository<Entity>` через `@InjectRepository`.

Допустимые базовые операции:

- `find`
- `findOne`
- `findAndCount`
- `insert`
- `update`
- `save`

Правила работы с БД:

- использовать типизированные сущности
- связи между сущностями описывать явно
- принадлежность данных владельцу учитывать в сервисах и запросах
- изменение схемы базы данных выполнять только через миграции

Настройки БД и `DataSource` должны находиться в `backend/src/database`.

---

## 5. DTO и трансформация ответов

DTO являются HTTP-границей приложения.

Правила:

- входные DTO валидируются через `class-validator`
- выходные DTO не содержат бизнес-логику
- DTO совместимы с контрактами из `backend/src/_contracts`
- контроллер получает результат от сервиса и трансформирует его в DTO через общую утилиту

Используется утилита `transformToDto` из [backend/src/_common/utils/transform-to-dto.ts](/Users/viig/www/teachly/backend/src/_common/utils/transform-to-dto.ts).

Контроллер должен:

- принять request DTO
- вызвать сервис
- вернуть response DTO

Контроллер не должен вручную собирать ответ, если его можно получить общей трансформацией.

## 6. Контракты

Слой контрактов расположен в `backend/src/_contracts`.

Назначение:

- описывать типы внешнего API через `interface`
- описывать API-контракты для фронтенда
- быть единым источником типов для backend и frontend
- собираться в отдельный npm-пакет

Структура:

```text
backend/src/_contracts/
  api-base.ts
  index.ts
  {module}/
    index.ts
    {entity}.type.ts
    create-{entity}.type.ts
    {module}-api.ts
```

Правила:

- backend DTO реализуют контракты через `implements`
- модульные `index.ts` реэкспортируют содержимое модуля
- корневой `index.ts` реэкспортирует все контракты
- пакет контрактов публикуется в GitHub Packages
- версия пакета обновляется через `.github/scripts/bump-version.sh`

## 7. Общие константы и утилиты

Магические значения в коде не допускаются.

Правила:

- константы модуля хранить в `backend/src/{module}/constants/index.ts`
- общие типы хранить в `backend/src/_common/types`
- общие утилиты хранить в `backend/src/_common/utils`
- общую инфраструктуру приложения хранить в `backend/src/_common/app`

Правила для текстов и локализации:

- пользовательские тексты не должны бесконтрольно хардкодиться в бизнес-логике
- тексты интерфейса и шаблонов уведомлений должны быть готовы к вынесению в словари локализации
- коды статусов, типов и системных значений должны храниться как стабильные константы, независимо от языка отображения

Правила для шаблонов уведомлений:

- для формирования уведомлений используется `Handlebars`
- шаблоны уведомлений должны храниться в файловой системе, а не вшиваться в код
- структура хранения шаблонов должна допускать разбиение по каналам, типам уведомлений и языкам
- локализация уведомлений должна решаться выбором шаблона по языку, а не ветвлением строк в бизнес-логике

## 8. Доступ к данным

Для MVP используется простая модель доступа:

- основная роль: преподаватель
- каждый пользователь видит только свои данные
- все доменные сущности должны быть привязаны к владельцу

Правила:

- проверки принадлежности данных не должны жить только в контроллерах
- фильтрация по владельцу должна учитываться в сервисах и, при необходимости, в репозиториях
- доступ к чужим `students`, `groups`, `lessons`, `payments`, `notifications` запрещен

## 9. Доменные правила MVP

### `lesson` и `payment`

Для MVP используется простая финансовая модель:

- `scheduled` занятие не влияет на баланс
- `completed` занятие формирует начисление
- `cancelled` занятие не формирует начисление
- `rescheduled` означает перенос существующего занятия и не является отдельным финансовым событием
- `payment` фиксирует факт поступления денег

Рекомендуемая логика:

- баланс ученика = сумма оплат минус сумма начислений по проведенным занятиям
- задолженность = `max(0, начисления - оплаты)`
- положительный баланс означает аванс или переплату

В MVP не нужно жестко связывать один `payment` с одним `lesson`. Баланс считается агрегированием начислений и оплат.

### Повторяющиеся занятия

Повторяющиеся занятия нужно моделировать как:

- шаблон серии
- отдельные материализованные экземпляры занятий

Финансовый учет, посещаемость, переносы и отмены должны жить на уровне конкретного экземпляра занятия.

## 10. Тестирование

Обязательные уровни тестов:

- unit tests
- e2e tests

### Unit tests

- тестируют бизнес-логику
- внешние зависимости и репозитории мокаются

### e2e tests

- выполняются в изолированной среде
- используют отдельный `PostgreSQL` в `docker-compose`
- база тестов работает на `tmpfs`
- тесты не должны зависеть от локального Postgres разработчика

Правила e2e:

- перед запуском тестов применяются миграции
- перед каждым тестом таблицы очищаются
- каждый тест создает только нужные ему данные
- для подготовки данных используются фабрики и общие утилиты

Переиспользуемые тестовые утилиты:

- очистка таблиц
- создание тестового приложения и HTTP server
- получение `Repository<Entity>` в тесте

Тесты и миграции должны запускаться в CI при push и при merge в `main`.
