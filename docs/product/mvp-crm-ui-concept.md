# UI-концепция MVP CRM

Документ фиксирует визуальное направление этапа 2 и служит основой для согласования до начала этапа 3.

Этап 2 пока не считается завершенным, пока визуальный подход и базовые экраны не согласованы.

## 1. Текущее направление

Референс: https://flux-dashboard.pages.dev/
Пожелание использовать фреймоврк `daisyUI` в реализации.

Это не означает копирование чужого CRM один в один. Референс используется как ориентир по:

- общему качеству UI-shell
- плотности и дисциплине отступов
- иерархии экранов
- аккуратности таблиц, форм и action-паттернов
- ощущению "готового продукта", а не концепта

Для Teachly выбран следующий вектор:

- светлая нейтральная база без перегруженных декоративных фонов
- строгий desktop shell: sidebar + topbar + контентная область
- компактная, но читаемая навигация
- акцентный синий для primary actions
- `daisyUI-first` как базовый слой визуального языка
- формы, таблицы и панели выглядят как части одной продуктовой системы

## 2. Визуальные принципы

- интерфейс должен выглядеть как рабочий продукт преподавателя, а не как презентационный concept mock
- визуальная система должна быть спокойной, но собранной
- плотность интерфейса должна позволять быстро сканировать данные
- главное внимание отдается структуре, читаемости и сценарию работы, а не декоративным эффектам
- главные действия должны быть заметны сразу, вторичные не должны спорить с ними

Что сознательно исключается:

- лишняя "атмосферная" декоративность
- большие рекламные hero-блоки
- концептуальные описательные тексты вместо реальных подписей
- случайные кастомные радиусы и нестабильные авторские паттерны поверх `daisyUI`

## 3. DaisyUI-first

Для этапа 2 зафиксирован принцип:

- по максимуму использовать готовые компоненты `daisyUI`
- не переопределять радиусы вручную там, где это уже решено самим компонентом
- кастомный UI допустим только там, где нужного паттерна нет в `daisyUI`
- даже в кастомных блоках нужно сохранять стилистику `daisyUI`

Базовые компоненты, на которые должен опираться интерфейс:

- `btn`
- `card`
- `input`
- `select`
- `textarea`
- `table`
- `badge`
- `menu`
- `dock`
- `navbar`
- `tabs`

Тема:

- используется встроенная тема `corporate` как ближайшая к нужному нейтральному SaaS-направлению
- кастомизация допустима точечно, без превращения темы в полностью авторский UI kit

## 4. Продуктовая логика интерфейса

Интерфейс должен отражать реальный рабочий процесс преподавателя:

- ученики и группы являются базой для дальнейших действий
- календарь остается центральным инструментом, но не вытесняет остальные сущности
- запись в календаре создается из карточки ученика или группы
- профиль ученика и профиль преподавателя должны быть не read-only summary, а редактируемыми формами
- финансовая часть показывает долг и историю оплат, но не превращается в бухгалтерский интерфейс

Из этого следуют важные UX-правила:

- глобальные кнопки не должны противоречить продуктовому сценарию
- основные действия живут там, где пользователь ожидает их найти
- если сущность имеет профиль, у нее должен быть явный вход в детальный экран
- детальный экран должен давать возможность редактирования, а не только просмотра

## 5. App Shell

Новый shell этапа 2 фиксируется так:

- desktop: левый sidebar, верхний topbar, отдельная основная зона контента
- sidebar компактный, с иконками, короткими подписями и вторичными hints
- topbar содержит дату, глобальный поиск и уведомления
- глобальные CTA не должны без причины висеть на всех страницах
- page header на экране отвечает за локальные действия конкретной сущности

Цель shell:

- дать ощущение единой системы
- обеспечить быстрый переход между основными разделами
- не перегружать экран повторяющимися action-блоками

## 6. Формы, таблицы и карточки

Основной акцент нового направления:

- формы и таблицы важнее, чем "декоративные карточки"
- данные должны выглядеть как реальные рабочие данные
- таблицы на desktop должны быть строгими и компактными
- карточки на mobile должны заменять таблицы без потери смысла

Правила для форм:

- профиль ученика редактируется через полноценную форму
- профиль преподавателя редактируется через полноценную форму
- рабочие интервалы преподавателя задаются специальными controls выбора времени
- для интервалов используется шаг 15 минут

Правила для списков:

- в desktop-версии список учеников и похожие сущности должны читаться как продуктовые data tables
- в mobile-версии эти же данные раскладываются в карточки с явными действиями
- важные сигналы вроде долга или статуса не должны теряться в подпоясных текстах

## 7. Mobile-first как обязательное требование

Часть преподавателей будет работать с телефона, поэтому mobile-friendly входит в обязательный фундамент, а не в пост-адаптацию.

Это означает:

- мобильная навигация должна быть постоянной и быстрой
- content area не должна перекрываться fixed dock
- ключевые действия должны быть достижимы одной рукой
- сложные таблицы на телефоне заменяются карточками
- формы должны оставаться читаемыми и управляемыми на маленьком экране

Для мобильного слоя уже зафиксированы отдельные правила:

- используется нижний `dock`
- последний tab отражает реальную сущность `Профиль`
- под fixed dock всегда должен резервироваться отдельный layout-space
- mobile UI должен быть полноценным рабочим интерфейсом, а не урезанной копией desktop

## 8. Что уже отражено в mock-слое

Во `frontend/` уже заложены и частично согласованы:

- `Nuxt 4`
- `Tailwind CSS v4`
- `daisyUI v5`
- desktop shell в новой более строгой стилистике
- mobile dock и mobile-safe layout
- список учеников в более зрелом SaaS-стиле
- профиль ученика как редактируемая форма
- профиль преподавателя как редактируемая форма

Текущий визуальный вектор лучше всего описывается так:

- ближе к product dashboard
- дальше от "concept card UI"
- меньше декоративности
- больше системности

## 9. Что еще требуется довести в этапе 2

До окончательного согласования этапа 2 нужно привести к тому же стилю:

- группы
- календарь
- оплаты
- overview
- настройки и вспомогательные экраны в полном визуальном единстве

Также нужно проверить:

- согласованность desktop и mobile
- единый уровень плотности между экранами
- достаточную читаемость таблиц и форм
- отсутствие нереалистичных действий и текстов

## 10. Критерии согласования этапа 2

Этап 2 можно считать согласованным только если:

- визуальный язык продукта стабилен и последователен
- ключевые экраны выглядят как части одной системы
- mobile-friendly реализован как полноценный сценарий
- основные формы и рабочие экраны выглядят реалистично
- дальнейший этап 3 может строиться уже не на поиске стилистики, а на переносе согласованного UI на все mock-экраны
