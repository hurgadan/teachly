# План реализации MVP CRM для частных преподавателей иностранных языков

Этот документ описывает этапы реализации продукта, очередность работ и текущий прогресс.

Бизнес-требования, основная модель продукта и пользовательские сценарии описаны в [docs/product/mvp-crm-plan.md](/Users/viig/www/teachly/docs/product/mvp-crm-plan.md).

---

## 1. Правила отметки прогресса

Для наглядного отслеживания прогресса каждый этап и подэтап отмечается одним из двух состояний:

- `[ ]` не реализовано
- `[x]` реализовано

---

## 2. Этапы реализации

### [x] Этап 1. Проектирование

Цель этапа:

- зафиксировать продуктовую модель
- описать основной сценарий работы преподавателя
- определить ключевые сущности
- определить правила формирования расписания
- определить правила учета оплат

Результат этапа:

- понятное описание MVP
- зафиксированный основной флоу преподавателя
- согласованные бизнес-правила продукта

Подэтапы:

- [x] зафиксировать назначение продукта и аудиторию
- [x] определить границы MVP
- [x] зафиксировать основной сценарий работы преподавателя
- [x] определить правила формирования расписания
- [x] определить базовые правила оплат и задолженности

### [ ] Этап 2. UI-концепция и визуальный стиль

Цель этапа:

- определить визуальный язык продукта до начала полноценной реализации логики

Что должно появиться:

- [ ] общий визуальный стиль продукта
- [ ] типографика
- [ ] цветовая система
- [ ] базовые UI-компоненты
- [ ] HTML + CSS для всех основных экранов MVP
- [ ] HTML + CSS для ключевых состояний экранов и компонентов

Результат этапа:

- готовый визуальный фундамент продукта
- согласованный внешний вид основных экранов
- понимание будущего UX до начала backend-интеграции

### [ ] Этап 3. Кликабельный mock-интерфейс

Цель этапа:

- дать возможность оценить интерфейс и пользовательские сценарии до реализации бизнес-логики

Что должно появиться:

- [ ] все основные экраны MVP в frontend
- [ ] переходы между экранами
- [ ] замоканные кнопки и действия
- [ ] замоканные данные для демонстрации сценариев
- [ ] проверка пользовательского пути по основным сценариям

Результат этапа:

- можно оценить UI
- можно оценить UX
- можно быстро обнаружить проблемы в навигации, структуре экранов и сценариях

### [ ] Этап 4. Технический фундамент

Цель этапа:

- подготовить основу для реализации первой версии продукта
- запустить минимальный каркас системы

На этом этапе в работу идут только те части системы, которые нужны немедленно:

- авторизация
- пользователь преподавателя
- базовая основа frontend и backend

Принцип этапа:

- не создавать домены заранее "про запас"
- добавлять новый домен только тогда, когда он нужен следующему продуктовому этапу

Подэтапы:

- [ ] подготовить базовую структуру backend
- [ ] подготовить базовую структуру frontend
- [ ] подготовить окружение для поэтапной разработки

### [ ] Этап 5. Авторизация и аккаунт преподавателя

Цель этапа:

- дать преподавателю возможность зарегистрироваться
- создать профиль преподавателя
- сохранить базовые рабочие настройки

Что должно появиться:

- регистрация
- вход
- выход
- восстановление пароля
- профиль преподавателя
- рабочие настройки календаря
- базовая языковая настройка

Подэтапы:

- [ ] регистрация и вход
- [ ] профиль преподавателя
- [ ] настройки рабочего времени
- [ ] настройки буфера после занятия
- [ ] базовая языковая настройка

### [ ] Этап 6. Ученики и группы

Цель этапа:

- дать преподавателю возможность вести свою клиентскую базу

Что должно появиться:

- карточка ученика
- карточка группы
- добавление участников в группу
- указание цены ученика
- указание длительности занятий ученика или группы
- статусы и комментарии

Подэтапы:

- [ ] создание и редактирование ученика
- [ ] создание и редактирование группы
- [ ] управление составом группы
- [ ] хранение цены ученика
- [ ] хранение длительности занятия

### [ ] Этап 7. Календарь и расписание

Цель этапа:

- реализовать центральный рабочий инструмент преподавателя

Что должно появиться:

- общий календарь преподавателя
- выбор доступных слотов
- создание расписания из карточки ученика
- создание расписания из карточки группы
- поддержка регулярных занятий
- поддержка разовых занятий
- учет буфера после занятия

Подэтапы:

- [ ] общий календарь преподавателя
- [ ] календарь доступных слотов
- [ ] создание расписания из карточки ученика
- [ ] создание расписания из карточки группы
- [ ] поддержка регулярных занятий
- [ ] поддержка разовых занятий

### [ ] Этап 8. Фиксация занятий и учет оплат

Цель этапа:

- связать календарь с фактической работой преподавателя и деньгами

Что должно появиться:

- отметка проведения занятия
- история занятий
- ручная фиксация оплаты
- отображение задолженности по ученику
- история оплат

Подэтапы:

- [ ] отметка проведения занятия
- [ ] история занятий
- [ ] ручная фиксация оплаты
- [ ] отображение задолженности по ученику

### [ ] Этап 9. Уведомления

Цель этапа:

- снизить долю ручной коммуникации

Что должно появиться:

- уведомление об отмене
- уведомление о переносе
- напоминание об оплате

Подэтапы:

- [ ] уведомление об отмене занятия
- [ ] уведомление о переносе занятия
- [ ] напоминание об оплате

### [ ] Этап 10. Ограничения и монетизация

Цель этапа:

- подготовить продукт к платному запуску

Что должно появиться:

- правила доступа к дополнительным возможностям

## 3. Приоритеты разработки

Первая очередь:

1. проектирование
2. UI-концепция и визуальный стиль
3. кликабельный mock-интерфейс
4. аккаунт преподавателя
5. ученики и группы
6. календарь и расписание
7. фиксация занятий
8. базовый учет оплат

Вторая очередь:

1. уведомления
2. тарифы и ограничения
3. улучшение UX календаря
4. история действий

Третья очередь:

1. роль ученика
2. кабинет ученика
3. интеграции с внешними календарями
4. ссылки на оплату
5. аналитика
6. публичная страница преподавателя
